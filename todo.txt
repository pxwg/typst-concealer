Phase 1 delivery status:

- Full overlay rendering is machine-owned: scans enter the reducer, candidates
  carry immutable request/page/version/scope identity, and old visible overlays
  retire only after the replacement commits.
- Hover conceal / unconceal transient state is routed through machine.runtime.
  The legacy read model still exists as a compatibility projection for display
  lookup code.
- Live preview now carries machine request identity and stale preview pages are
  rejected before apply.accept_page_update can run. A dedicated preview reducer
  remains a phase 2 cleanup, not a phase 1 blocker.
- Cursor/mode/text-change entry points route through machine.runtime wrappers.
- Image/extmark allocation and release for machine/full items is centralized in
  machine.resources; apply._new_image_id is only a compatibility shim.
- Fixed output path mitigation is implemented for machine requests: replacing a
  request abandons the old one, clears max(old,new) output pages, resets
  page_state, forces the next input write, and poll/ready callbacks only accept
  the active current_request.

Residual delivery risk:

- A single watch process with a fixed output prefix still cannot cryptographically
  prove that a physical PNG was generated by the current logical request. The
  current mitigation is covered by unit tests, but real continuous-edit watch
  testing should remain part of release validation. If stale physical pages are
  observed in practice, the next hardening step is request-scoped output paths.
